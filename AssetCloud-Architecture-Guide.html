💡 Design Principles

API-First & Event-Driven
Every capability exposed as a versioned REST API. Domain events published to Azure Event Grid for loose coupling between bounded contexts. ERP integrations consume the same public API surface.
Multi-Tenant by Design
Logical tenant isolation at data, compute, and network layers. Tenant-aware identity via Azure AD B2C with custom policies. Per-tenant encryption keys in Azure Key Vault (BYOK supported).
Composable Microservices
Domain-driven decomposition into bounded contexts (Origination, Fleet, Asset, Accounting, Remarketing). Each service owns its data and publishes domain events. No shared databases between services.
Regulatory-Native
GDPR, DORA, EBA guidelines, and IFRS 16 built into the platform from day one — not bolted on. Data residency controls, ICT resilience testing hooks, and audit trails are first-class concerns.

📊 Key Differentiators

AI-Powered Residual Value Engine
Combines historical transaction data, real-time telematics (mileage, condition scores), and market comparable feeds to predict residual values with high accuracy — reducing end-of-term write-offs.
Universal ERP Integration Hub
Pre-built adapters for Alfa Systems, Solifi/LeasePak, Linedata Ekip, SAP S/4HANA, FIS, and Sofico Miles. Adapter SDK enables custom connectors. Canonical data model abstracts vendor differences.
Telematics Intelligence Layer
IoT ingestion pipeline for OBD-II, CAN bus, and OEM telematics APIs. Powers predictive maintenance, driver behaviour scoring, fleet utilisation analytics, and insurance risk models.
Embedded Compliance Engine
Automated DORA ICT risk assessments, GDPR data subject request workflows, EBA outsourcing register generation, and continuous compliance monitoring with Azure Policy and Defender for Cloud.

🔌 Integration Architecture Pattern

The hub uses a Canonical Data Model + Adapter pattern. Core services speak only the canonical model. Each ERP adapter handles protocol translation (REST, SOAP, file), data mapping, error handling, and retry logic. Adapters run as isolated Azure Functions behind the Integration Service Bus, ensuring one vendor's issues do not cascade.

Compliance & Security Architecture

Regulatory compliance is embedded at every layer — not bolted on. The platform addresses DORA, GDPR, EBA guidelines, IFRS 16, and national regulatory requirements across EU member states.

Digital Operational Resilience Act (DORA)

Regulation (EU) 2022/2554 — Effective January 2025

ICT risk management framework, incident reporting, resilience testing, and third-party risk management. AssetCloud implements automated ICT risk assessments, disaster recovery testing (Azure Site Recovery), and a machine-readable third-party risk register.

HIGH IMPACT — Core architectural driver

General Data Protection Regulation (GDPR)

Regulation (EU) 2016/679

Data subject rights automation (access, erasure, portability), consent management, data retention policies with automated purge, and privacy-by-design. Data residency enforced at Azure region level (West Europe / North Europe).

HIGH IMPACT — Data architecture driver

EBA Outsourcing Guidelines

EBA/GL/2019/02

Cloud outsourcing risk assessment, exit strategy documentation, audit access provisions, and sub-outsourcing chain transparency. AssetCloud maintains an outsourcing register with Azure service dependency mapping.

MEDIUM IMPACT — Operational governance

IFRS 16 — Leases

International Financial Reporting Standard 16

Right-of-use asset and lease liability recognition, modification accounting, variable lease payment handling, and disclosure generation. Multi-GAAP support for institutions reporting under both IFRS and local GAAP.

HIGH IMPACT — Accounting engine design

PSD2 / Open Banking

Directive (EU) 2015/2366

Strong Customer Authentication (SCA) for payment initiation, account information APIs following Berlin Group NextGenPSD2, and TPP registration validation.

MEDIUM IMPACT — Payment flows

EU AI Act

Regulation (EU) 2024/1689

Credit scoring classified as high-risk AI system. Requires: risk management system, data governance, transparency documentation, human oversight, and conformity assessment. AssetCloud implements model cards, SHAP explainability, and fairness monitoring.

Key Architecture Decisions

Architecture Decision Records (ADRs) documenting the critical choices shaping AssetCloud.

ADR-001: Azure Kubernetes Service as Primary Compute Runtime

Accepted

Context: Need a container orchestration platform that supports microservices, event-driven scaling, service mesh, and multi-region deployment while aligning with Azure-first strategy.

Decision: AKS with Dapr sidecar for service invocation, pub/sub, and state management. KEDA for event-driven auto-scaling. Flux CD for GitOps-based deployments.

Rationale: AKS provides the deepest Azure integration (AAD pod identity, Key Vault CSI, Azure Monitor) while maintaining Kubernetes portability. Dapr abstracts infrastructure concerns, making it feasible to run on other Kubernetes distributions if cloud strategy evolves. Considered Azure Container Apps but it lacks the fine-grained control needed for financial services workload isolation and compliance.

ADR-002: Canonical Data Model for ERP Integration

Accepted

Context: Must integrate with 6+ ERP vendors, each with different data models, protocols, and semantics. Direct point-to-point integration creates O(n²) complexity.

Decision: Define a canonical data model (CDM) for AssetCloud domain entities. Each ERP adapter maps to/from the CDM. Core services only interact with the CDM.

Rationale: Reduces integration complexity from O(n²) to O(n). Adding a new ERP vendor requires only one adapter, not changes across all consuming services. Trade-off: the CDM must be expressive enough to cover all vendor capabilities without becoming a lowest-common-denominator. Mitigated by extension points in the CDM schema and vendor-specific metadata bags.

ADR-003: Event Sourcing for Origination & Payments

Accepted

Context: Audit trail requirements for financial transactions are strict (DORA, EBA). Need immutable record of every state change with ability to reconstruct state at any point in time.

Decision: Event-sourced architecture for Origination and Payments bounded contexts using Cosmos DB as the event store. CQRS pattern with dedicated read models in Azure SQL.

Rationale: Event sourcing provides a complete, immutable audit trail by design — satisfying regulatory requirements without additional audit-logging infrastructure. CQRS enables optimised read models for different query patterns. Trade-off: increased complexity in event schema evolution and eventual consistency. Mitigated by schema registry and versioned event contracts.

ADR-004: Multi-Tenant with Logical Isolation

Accepted

Context: Serving multiple banks/finance companies from a shared platform. Some customers may require stronger isolation guarantees.

Decision: Default: logical multi-tenancy with row-level security, tenant-specific encryption keys, and network policies. Premium tier: dedicated AKS node pools and database instances per tenant.

Rationale: Logical isolation is cost-efficient and operationally simpler for most customers. The premium tier satisfies banks with stricter regulatory interpretations or internal policies. Azure Key Vault BYOK ensures each tenant controls their own encryption keys even in the shared model.

ADR-005: Azure OpenAI for Generative AI Capabilities

Accepted

Context: Need LLM capabilities for document understanding, conversational analytics, and natural-language fleet insights. Must satisfy EU data residency and AI Act requirements.

Decision: Azure OpenAI Service (Sweden Central region) with GPT-4o for generative tasks. Data stays within EU. Supplemented by Azure AI Document Intelligence for structured extraction.

Rationale: Azure OpenAI runs in EU data centres (Sweden Central), data is not used for model training, and enterprise security controls (private endpoints, managed identity) apply. The EU AI Act classifies credit scoring as high-risk — SHAP explainability and human oversight are built into the ML pipeline, not the LLM layer. LLMs handle advisory/conversational tasks that don't require the same conformity assessment.

ADR-006: Terraform for Infrastructure as Code (with Bicep for Azure-Native Modules)

Proposed

Context: Azure-first but want to preserve portability option. Team has mixed Terraform/Bicep experience.

Decision: Terraform as primary IaC with Azure Verified Modules. Bicep for Azure-specific complex resources (e.g., AKS cluster config, APIM policies) wrapped as Terraform modules.

Rationale: Terraform provides multi-cloud portability and a mature state management model. Azure Verified Modules reduce custom code. Bicep handles the 20% of Azure resources where HCL is cumbersome. Trade-off: two IaC languages increase onboarding complexity. Mitigated by clear boundaries (Terraform for orchestration, Bicep for Azure modules).

Delivery Roadmap

Phased delivery over 18 months, starting with core platform and origination, then expanding to fleet lifecycle and AI capabilities.

Phase 0 — Foundation (Months 1–3)

Platform & Infrastructure Bootstrap

Azure Landing Zone deployment (FSI pattern), AKS clusters, CI/CD pipelines, observability stack, identity platform, and IaC framework. Security baseline and compliance posture established. Integration Hub skeleton with first adapter (Solifi) in sandbox.

Phase 1 — Core Origination (Months 3–6)

Vehicle Finance Origination MVP

Dealer portal, quote engine, credit application workflow, automated credit scoring (v1), document collection (Document AI), approval workflow, and contract generation. Alfa Systems and SAP adapters in development. IFRS 16 accounting engine (basic).

Phase 2 — Fleet Lifecycle (Months 6–10)

In-Service Fleet Management

Fleet manager portal, vehicle ordering, driver management, maintenance scheduling, telematics gateway (IoT Hub), and mobile driver app (v1). Predictive maintenance model (v1). Sofico Miles and FIS adapters. SEPA payment integration.

Phase 3 — AI & Analytics (Months 10–14)

Advanced AI Capabilities & Data Platform

Residual value prediction model, fraud detection engine, fleet optimisation AI, conversational analytics (Azure OpenAI), portfolio analytics dashboard (Synapse), and regulatory reporting automation. Linedata Ekip adapter. Full MLOps pipeline.

Phase 4 — Remarketing & Scale (Months 14–18)

End-of-Term & Market Expansion

Remarketing module (auction integration, channel optimisation AI), advanced driver behaviour scoring, insurance integration, Open Banking APIs (PSD2), multi-country rollout (DE, FR, NL, Nordics), and premium tenant isolation tier. Full compliance certification.
